## Overview

Execute one full launch-attempt loop for the next available candidate under the post-PR-#64 reliability-fixed runtime. The primary regression gate is CP1: `runtime_heartbeat_age_seconds < 60` and `guard_snapshot_age_seconds` populated within 2 minutes — confirming the April 14 failure mode is resolved. Then run the full 30-minute z-contract monitoring protocol and produce auditable closure evidence.

### What if we didn't do this step?

**Immediate Impact:**
The PR #64 reliability fixes (guard-snapshot retry loop, lock-timeout guards, commit-before-loop) remain unvalidated in production. We cannot confirm the April 14 `runner_liveness_failed` failure mode is resolved.

**Downstream Consequences:**
Any follow-on infrastructure work proceeds without a verified live baseline, increasing the risk that the next launch fails for the same unresolved reason or a related contention issue.

**Technical Debt:**
The April 14 failure evidence remains the last canonical launch record. Without a confirmed-success (or a new identified failure) closure, regressions and improvements cannot be distinguished.

**Alternative Approaches:**
We could defer launch until additional unit tests pass, but the retry and lock-timeout paths require a live session to exercise; synthetic tests alone are insufficient validation.

**Risk Assessment:**
High — the April 14 failure blocked all live trading for over a week. Validating the fix is the critical path to resuming live operations.

## Requirements

1. Record candidate ID, intended notional, and rationale for selection before launch.
2. Run preflight checks and confirm all services healthy:
   - `poetry run trade ops status --registry configs/services/registry.prod.yaml --format table --tag core`
   - `poetry run trade ops status --registry configs/services/registry.prod.yaml --format table --tag scanner`
   - Confirm Scanner Status shows the `e206` consolidated scanner with fresh refresh age.
3. Start session via candidate launch command and capture returned session ID:
   ```
   PRODUCTION_TRADING_ENABLED=1 poetry run trade live start-candidate \
     --candidate-id <candidate_id> --initial-capital <capital> --yes
   ```
4. **CP1 regression gate** (≤2 minutes post-launch) — log explicitly before continuing:
   - `runtime_heartbeat_age_seconds < 60` ✓ or ✗
   - `guard_snapshot_age_seconds` populated (not `None`) ✓ or ✗
   - `health_gate` status
   - If CP1 fails the regression gate: stop immediately, log evidence, and record as a new blocker (do not continue to CP2).
5. Run monitoring for 30 minutes at 1-minute intervals (operator may approve a different cadence; deviation must be logged in evidence) using the default z-contract fields:
   - **Flat lane (no open position):** `z_score`, `z_entry` threshold, expected gross edge vs trigger, orders/fills/open-position counters.
   - **Open-position lane:** `z_score`, `z_exit` threshold, pair gross MTM delta, gross flatten thresholds (take-profit/stop-loss), orders/fills/open-position counters.
   - If any checkpoint is missing required z-contract fields, log it as a monitoring-contract breach and note the remediation action.
6. If any manual intervention occurs (stop/emergency-stop/manual flatten), log timestamp, reason, and observed evidence.
7. Publish a short closure note with:
   - launch outcome (`normal`, `manually_stopped`, `emergency_stopped`, `blocked`, `runner_liveness_failed`)
   - CP1 regression gate result (`pass` / `fail`)
   - health-gate status at launch and during session
   - monitoring-contract compliance result
   - whether guard-snapshot retry or lock-timeout paths were exercised (check service logs)
   - follow-up fix recommendations (if needed)
