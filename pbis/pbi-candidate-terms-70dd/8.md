# 8 Adena solo through the new pipeline

Back to task list: `tasks.md`

## Overview

Run Adena through the candidate-terms pipeline end-to-end as the **first real exercise** of the new machinery. The pre-state (after Task 7) is: source vocabulary preserved, `semantic_modeling` empty, `core_semantics.canonical_concepts` reduced to seed roots only.

The expected outcome is that the canonical tree **stays at seed-roots-only** after Adena solo. Single-source promotions are blocked by the placement gate unless the operator explicitly invokes `--allow-root` with rationale, which should be rare for a healthcare-vocabulary single-source pass. Most Q8 rows should resolve to `defer`. This task validates that the machinery handles single-source correctly without growing the canonical language prematurely.

This task replaces the original "E2E validation against synthetic fixtures" plan. Synthetic-fixture validation already happened during the Wave 1 parallel-agent runs (Tasks 2–6 each ran their own probes). The remaining validation that has value is the real one: does the pipeline produce sensible decisions against Adena's actual vocabulary?

### What if we didn't do this step?

**Immediate Impact:** Task 9 would be the first real run, mixing single-source baseline behavior with cross-source convergence. We'd lose the ability to attribute observed behavior to the right cause.

**Downstream Consequences:** If single-source bugs exist (e.g., placement gate too lenient, deferral rationale not surfaced correctly), they'd compound with cross-source bugs and make Task 9 harder to interpret.

**Technical Debt:** Skipping the single-source case means we never demonstrated the "no orphan promotions" property against real data.

**Alternative Approaches:** Roll Task 8 directly into Task 9. Rejected because the value of Task 8 is observing what happens with `cross_source_count = 1` everywhere — a regime Task 9 doesn't visit cleanly.

**Risk Assessment:** Low — read-side validation; any writes are made through the new machinery itself, which has its own gates.

## Description

Execute the following sequence and capture results in the PR description (or task notes if no code changes are produced).

### Pre-checks

1. Confirm Task 7 post-state: `canonical_concepts` count = number of seed roots; `mapping_definitions` count = 0; `candidate_terms` count = 0.
2. Confirm Adena's source vocabulary is intact: `source_vocabulary_models`, `vocabulary_surfaces`, `vocabulary_terms` row counts match pre-purge snapshot.

### Recording

`.venv/bin/python3 peddlerr/workbench/scripts/record_candidate_terms.py --source-system adena-health-system`

Expected: a fresh batch of `candidate` rows for every Adena vocabulary term plus one observation each. Capture `created` / `re_observed` / `skipped` counts.

### Q8 baseline

Run Q8 over Adena's undermapped term set. Expected partition (predominantly):
- `never_seen` → 0 (recording just attached every term to a candidate).
- `candidate` → high count (this run's recording).
- All other buckets → 0.

Capture the bucket counts as the Task 8 baseline.

### Skill walkthrough

Walk `reconcile-canonical-language` end-to-end against Adena's Q8 output. For each candidate, make an explicit decision (`promote`, `merge`, `reject`, `defer`) with non-empty `evaluation_notes`. Expected distribution:
- `defer` — most candidates, since `cross_source_count = 1` and convergence isn't yet observable.
- `merge` — some, where the agent recognizes intra-Adena vocabulary collapses (e.g., "Cardiology Svc" → "Cardiology").
- `reject` — few, where the term is clearly not canonical-language material.
- `promote` — ideally zero. Any promotion this round must use `--allow-root` with documented rationale, since no concept has cross-source signal and no concept's parent placement exists yet outside the seed roots.

### Decision execution

`.venv/bin/python3 peddlerr/workbench/scripts/promote_term_mappings.py --decisions <file>`

Capture: count per decision; any heuristic warnings; any rejected input rows.

### Post-state capture

Snapshot:
- `canonical_concepts` count and any new rows (expected: only seed roots; any new row's existence triggers an audit note).
- `candidate_terms` count partitioned by status (`candidate`, `deferred`, `merged`, `rejected`, `promoted`).
- `candidate_term_observations` count.
- Q8 re-run over Adena's undermapped set — buckets should now reflect the made decisions.
