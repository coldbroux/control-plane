# 8 E2E validation of accumulating candidate set

## Overview

Run a full Phase 2 cycle (record → evaluate → Q8 review) against Adena (and ideally a second source if one is available) and confirm that prior rationale is surfaced, recording is idempotent, and Q8 buckets behave correctly across runs. Closes the PBI by exercising the full memory loop end-to-end.

### What if we didn't do this step?

**Immediate Impact:** PBI lands without a runtime confirmation that the pieces compose.

**Downstream Consequences:** Bugs surface during the next real Phase 2 run rather than now, when context is fresh.

**Technical Debt:** None — this is the validation step itself.

**Alternative Approaches:** Rely solely on per-task tests. Insufficient — they each test their own slice and miss integration issues (transaction boundaries, schema drift, skill output that doesn't match script input).

**Risk Assessment:** Low — runs against an isolated local DB.

## Description

Execute the following sequence and capture results in the PR description:

1. **Recording, run 1:** `record_candidate_terms.py --source-system adena-health-system`. Confirm Adena's vocabulary terms are now represented as candidates (post-backfill, this is mostly a no-op for previously-deferred terms — observations should already exist).
2. **Recording, run 2 (idempotency):** Re-run; confirm zero new rows.
3. **Q8 baseline:** Run Q8 over Adena's undermapped terms; capture bucket counts.
4. **Skill rehearsal:** Walk the reconcile-canonical-language skill against Q8 output (manually or via the slash command). Confirm the agent surfaces prior `evaluation_notes` for the 38 backfilled deferrals and proposes new evaluations only for `never_seen` rows.
5. **Promotion + merge round-trip:** Pick one candidate to promote and one to merge into another. Run `promote_term_mappings.py` with both decisions. Confirm:
   - The promoted candidate transitions to `promoted` with `resolved_concept_key` set.
   - The merged candidate transitions to `merged` with `merged_into_candidate_id` set.
   - `evaluation_notes` are populated.
   - The CHECK constraint did not fire.
6. **Q8 follow-up:** Re-run Q8. Confirm the promoted and merged candidates appear in the appropriate buckets and the deferred bucket has shrunk by exactly two.
7. **Cross-source convergence (if a second source is available):** Run recording for a second system that shares one or more terms with Adena. Confirm `cross_source_count` rises to 2 for those candidates and that the heuristic warning no longer fires for them.
8. **Obsolete sweep:** Bump Adena's vocabulary version (artificially flip `is_current` if needed in a test DB) and run `sweep_obsolete_candidates.py`. Confirm a deferred candidate whose only observation references the now-non-current version transitions to `obsolete`.
