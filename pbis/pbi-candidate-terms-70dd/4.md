# 4 `promote_term_mappings.py` — lifecycle evaluation

Back to task list: `tasks.md`

## Overview

Extend the existing `peddlerr/workbench/scripts/promote_term_mappings.py` so that every Phase 2 evaluation produces an explicit lifecycle transition on the relevant `candidate_terms` rows, with `evaluation_notes` populated. This is the "canonical-language decision" half of Phase 2; recording was handled by Task 3.

### What if we didn't do this step?

**Immediate Impact:** Candidate rows are created (Task 3) but never transition out of `candidate`. The lifecycle would stop at the entry state.

**Downstream Consequences:** Q8 buckets would be useless — every candidate would sit in `candidate` forever.

**Technical Debt:** Reasoning rationale would still be ephemeral.

**Alternative Approaches:** None — promotion/deferral/merge/rejection are inherent to Phase 2 by design.

**Risk Assessment:** Medium — touches the script that performs Brian's-language updates. Errors here can corrupt the canonical concept tree.

## Description

`promote_term_mappings.py` currently accepts an input file describing which terms to promote and writes mapping definitions. Extend it so:

1. **New input categories.** The input file (or new flags) supports four explicit operator decisions per term:
   - `promote` → existing behavior + transition the candidate to `status='promoted'`, set `resolved_concept_key` and `resolved_at`, populate `evaluation_notes`.
   - `merge` → set `status='merged'`, `merged_into_candidate_id`, `resolved_at`, populate `evaluation_notes`.
   - `reject` → set `status='rejected'`, populate `evaluation_notes`.
   - `defer` → set `status='deferred'`, populate `evaluation_notes`.
2. **Atomic transitions.** Each candidate transition is a single transaction including the canonical-language write (for `promote`) and the candidate row update.
3. **Notes are required.** All four decisions require a non-empty `evaluation_notes` string. The script rejects an input row that lacks one.
4. **Promotion heuristic surfaced, not gated.** Per PRD Open Question 1, v1 surfaces the heuristic check (≥2 sources OR widely-understood) as a warning, not a hard block. The operator can still promote a single-source candidate by providing explicit justification in `evaluation_notes`. (Update this requirement once the open question is settled.)
5. **Re-promotion of `merged` and `rejected` candidates is forbidden.** The script must reject input that targets a non-active candidate.
6. **Backwards compatibility.** Existing inputs that only ever specified promotions should continue to work, treated as `promote` decisions. This compatibility may be removed once Phase 2 always emits the new format.

## Requirements

1. The script supports four decisions: `promote`, `merge`, `reject`, `defer`.
2. Every decision requires a non-empty `evaluation_notes` string; the script errors out on missing notes.
3. Each candidate transition is performed in a single DB transaction.
4. Promotion writes the canonical-language artifact AND transitions the candidate row.
5. Merges set `merged_into_candidate_id` to a valid existing candidate id; the script validates the target exists and is in an active state.
6. Re-targeting a non-active candidate (`merged`/`rejected`/`obsolete`) is rejected with a clear error.
7. The promotion heuristic check produces a warning when fewer than 2 source systems have observed the candidate; the operator may still proceed with explicit justification.
8. Logged summary at end of run lists counts per decision and any warnings.
