# 9 Two-system E2E validation — canonical language convergence

Back to task list: `tasks.md`

## Overview

Run two AHRQ-compendium health systems through the full candidate-terms pipeline — Phase 1 (`model-source-semantics`) for both, then cross-system Phase 2 (`reconcile-canonical-language`) using the new recording, evaluation, and Q8 machinery. Verify that the canonical concept tree visibly reflects two-source provenance, and document either the convergence achieved or a structured analysis of why it was absent for the chosen pair.

This is the integration test of record. Task 8 verifies mechanics against fixtures; Task 9 verifies the system actually does what the PBI promises against real source vocabularies.

### What if we didn't do this step?

**Immediate Impact:** The PBI lands without a real-world demonstration that cross-source convergence happens. Acceptance criteria 1–11 can pass against synthetic data while the canonical-language outcome remains unproven.

**Downstream Consequences:** The first time the system is used for real, surprises emerge in production-equivalent conditions rather than during validation.

**Technical Debt:** Same as immediate impact — the integration risk is unmitigated.

**Alternative Approaches:** Wait for organic two-system runs to occur in the course of normal operation. Rejected — that defers risk and conflates validation with delivery.

**Risk Assessment:** Medium — depends on whether the chosen second system shares enough vocabulary with Adena for convergence to be observable. Pair selection must be deliberate.

## Description

Execute the following sequence and capture results in the PR description.

### Pair selection

Pick a second health system from the AHRQ compendium whose public vocabulary plausibly overlaps Adena's. Document the rationale for the choice (e.g. similar size, similar service-line breadth, similar geographic context). The pair selection is itself part of the task output — future operators should be able to reuse the heuristic.

### Phase 1 for system B

Run `model-source-semantics` end-to-end for the selected system: surface discovery, semantic config persistence, Neo4j sync, validation. Confirm the system has an `is_current` model version with a populated vocabulary.

(Phase 1 for Adena was completed during `pbi-research-orch-2e0b` Task 8 and is a prerequisite, not part of this task's scope.)

### Cross-system Phase 2

1. `record_candidate_terms.py --source-system adena-health-system` — should be a near-no-op since Adena observations were established by Task 7 backfill plus any earlier Task 3 runs.
2. `record_candidate_terms.py --source-system <system-b>` — creates new candidates for system B's vocabulary, attaches new observations to existing candidates where exact-string normalized form matches Adena's.
3. Run Q8 over the union of both systems' undermapped terms. Capture the bucket counts and the rows where `cross_source_count = 2`.
4. Walk the `reconcile-canonical-language` skill against Q8 output. Make explicit decisions for each row: `promote`, `merge`, `reject`, or `defer`. For at least one candidate where `cross_source_count = 2`, the decision must be `promote` if the heuristic supports it (this is the core validation: cross-source convergence drives canonical-concept growth).
5. Run `promote_term_mappings.py` with all decisions; confirm transitions are written.

### Canonical language verification

Capture and attach to the PR:

1. **Pre/post canonical concept count.** `SELECT count(*) FROM core_semantics.canonical_concepts;` before and after Task 9 promotions.
2. **Cross-source provenance query.** Identify canonical concepts whose source observations span both systems:
   ```sql
   SELECT cc.concept_key,
          ct.normalized_term,
          count(DISTINCT vt.source_system) AS source_system_count
   FROM core_semantics.canonical_concepts cc
   JOIN semantic_modeling.candidate_terms ct
     ON ct.resolved_concept_key = cc.concept_key
   JOIN semantic_modeling.candidate_term_observations o
     ON o.candidate_term_id = ct.candidate_term_id
   JOIN core_semantics.vocabulary_terms vt
     ON vt.id = o.vocabulary_term_id
   WHERE ct.status = 'promoted'
   GROUP BY cc.concept_key, ct.normalized_term
   HAVING count(DISTINCT vt.source_system) >= 2;
   ```
   (Final SQL form to be adjusted to actual column names.)
3. **Q8 partition snapshot.** The Q8 output post-decisions, showing the candidate set partitioned across the six lifecycle states with `cross_source_count` per row.
4. **Convergence analysis.** A short narrative: which terms converged, which did not, and what the chosen system pair's vocabulary overlap looked like.

### Absence-of-convergence handling

If no candidates reach `cross_source_count = 2`, do not force a promotion. Instead, document:

- The vocabulary overlap (or lack thereof) between Adena and the chosen system B.
- Whether the absence reflects truly disjoint vocabularies or a normalization gap (exact-string matching missed a real overlap).
- A recommendation for source pairing or normalization improvements that would make the next attempt more productive.

This outcome is a valid task completion. The PBI's value is the mechanism; the mechanism working against unhelpful inputs is still a successful demonstration.
