# 9 Adena + System B; demonstrate cross-source convergence

Back to task list: `tasks.md`

## Overview

Build on Task 8's clean post-Adena-solo state. Phase-1-and-record System B as a second source. Walk the candidate-terms pipeline again — now with cross-source signal available — and observe canonical concepts emerging from convergence between Adena and System B vocabularies.

This is the integration test of the PBI's value proposition: the new machinery, when fed two source vocabularies that share clinical/operational language, produces canonical concepts whose provenance is traceable to multi-source observations and explicit operator decisions.

The success outcome is that **every non-seed canonical concept post-Task-9** has:
1. A `candidate_terms` row referencing it via `resolved_concept_key`,
2. With non-empty `evaluation_notes` documenting the promotion rationale,
3. With ≥2 observations from distinct source systems (or, if `--allow-root` was used, an explicit and documented operator rationale).

If no convergent terms are found, that's also a valid outcome — but it must come with a documented analysis of pair selection, vocabulary overlap, and normalization gaps.

### What if we didn't do this step?

**Immediate Impact:** The PBI ships without proving that cross-source convergence actually drives canonical-language growth in practice.

**Downstream Consequences:** The first time the system is used for real, behavior surprises emerge in production-equivalent conditions instead of during validation.

**Technical Debt:** Same as immediate impact — the integration risk is unmitigated.

**Alternative Approaches:** Synthetic two-source fixtures. Rejected — they cannot exercise pair-selection heuristics, real normalization gaps, or operator-decision discipline against actual healthcare vocabularies.

**Risk Assessment:** Medium — depends on whether the chosen System B has enough vocabulary overlap with Adena for convergence to be observable. Pair selection is itself part of the task.

## Description

Execute the following sequence and capture results in the PR description (or Task 9 notes).

### Pair selection

Pick a second AHRQ-compendium health system whose published vocabulary plausibly overlaps Adena's (similar size / service-line breadth / geographic context). Document the rationale for the choice. The pair-selection heuristic itself is part of the task output — future operators should be able to reuse it.

### Phase 1 for System B

Run `model-source-semantics` end-to-end for the selected system: surface discovery, semantic config persistence, Neo4j sync, validation. Confirm the system has an `is_current` semantic_model_version with a populated vocabulary.

(Phase 1 for Adena was completed during `pbi-research-orch-2e0b`; it is a prerequisite, not in this task's scope.)

### Two-source recording

1. `.venv/bin/python3 peddlerr/workbench/scripts/record_candidate_terms.py --source-system adena-health-system` — should be idempotent against Task 8's recording; zero new rows.
2. `.venv/bin/python3 peddlerr/workbench/scripts/record_candidate_terms.py --source-system <system-b>` — creates new candidates for System-B-only terms; existing candidates (where exact normalized form matches Adena) gain a second observation. After this step, any candidate with `cross_source_count = 2` is a convergence target.

### Q8 review

Run Q8 over the union of both systems' undermapped term sets. Capture the partition. Pay attention to:
- `deferred` rows where `cross_source_count` advanced from 1 to 2 (these are the highest-priority promotion candidates).
- `candidate` rows newly created by System B's recording.
- Any `merged`/`rejected`/`promoted` rows from Task 8 that should remain at their prior status.

### Skill walkthrough — cross-source pass

Walk `reconcile-canonical-language` over Q8 output. Promotion decisions for convergent candidates require:
- `parent_concept_key` (a seed root or other already-promoted parent) to satisfy the placement gate.
- Non-empty `evaluation_notes` capturing why this candidate earned promotion at this point.

For non-convergent rows, decisions remain mostly `defer` (single-source still). Some `merge` decisions may be appropriate for cross-source synonyms detected by the agent.

### Decision execution

`.venv/bin/python3 peddlerr/workbench/scripts/promote_term_mappings.py --decisions <file>`

Capture: count per decision; any heuristic warnings; any rejected inputs (especially placement-gate rejections).

### Provenance verification

For every non-seed canonical concept that exists post-execution, run:

```sql
SELECT cc.concept_key,
       cc.display_name,
       ct.normalized_term,
       ct.evaluation_notes,
       count(DISTINCT smv.source_namespace) AS source_system_count,
       array_agg(DISTINCT smv.source_namespace) AS source_systems
  FROM core_semantics.canonical_concepts cc
  JOIN semantic_modeling.candidate_terms ct
    ON ct.resolved_concept_key = cc.concept_key
  JOIN semantic_modeling.candidate_term_observations o
    ON o.candidate_term_id = ct.candidate_term_id
  JOIN core_semantics.semantic_model_versions smv
    ON smv.id = o.version_id
 WHERE cc.parent_concept_id IS NOT NULL
   AND ct.status = 'promoted'
 GROUP BY cc.concept_key, cc.display_name, ct.normalized_term, ct.evaluation_notes;
```

Expected: every row has `source_system_count >= 2`, OR has an `evaluation_notes` rationale that explicitly documents the `--allow-root` justification.

### Absence-of-convergence handling

If no candidates reach `cross_source_count = 2`, do not force a promotion. Document:
- The vocabulary overlap (or lack thereof) between Adena and System B.
- Whether the absence reflects truly disjoint vocabularies or a normalization gap (exact-string matching missed a real overlap).
- A recommendation for source pairing or normalization improvements that would make the next attempt more productive.

This outcome is a valid task completion. The PBI's value is the mechanism; the mechanism behaving correctly against unhelpful inputs is still a valid demonstration.
