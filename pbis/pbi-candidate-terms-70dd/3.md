# 3 `record_candidate_terms.py` — observation recording

Back to task list: `tasks.md`

## Overview

Author a new script `peddlerr/workbench/scripts/record_candidate_terms.py` that runs at the start of Phase 2. It reads the current `semantic_model_versions` row for a target source, walks its `vocabulary_terms`, and ensures each term is represented as an observation linked to a candidate. Idempotent: safe to re-run on the same input.

This script owns the "memory update" half of Phase 2. It does not evaluate or transition state — it only records observations and keeps `last_seen_version_id` current.

### What if we didn't do this step?

**Immediate Impact:** Candidate rows would have to be created as a side effect of `promote_term_mappings.py`, conflating recording with evaluation.

**Downstream Consequences:** Re-runs of the promotion script would re-do recording work; failures during evaluation could leave the candidate set in an inconsistent state.

**Technical Debt:** Mixing concerns in one script.

**Alternative Approaches:** Extend `promote_term_mappings.py` to do both. Rejected because the responsibilities are conceptually different — memory update vs. canonical-language decision.

**Risk Assessment:** Low — write-only, idempotent, no canonical-language impact.

## Description

The script accepts a `--source-system` (or `--namespace`) argument identifying the source whose vocabulary should be recorded. For each `vocabulary_terms` row in the current model version:

1. **Lookup:** Check `candidate_term_observations` for `(vocabulary_term_id, version_id)`. If present, no-op (idempotent).
2. **Match-or-create:** Compute the normalized form for the term. Look for an existing `candidate_terms` row with that exact `normalized_term` AND `status IN ('candidate', 'deferred', 'promoted')`.
   - If found → reuse: insert observation, set `last_seen_version_id` on the candidate, no status change.
   - If not found → create a new candidate row with `status='candidate'`, `first_seen_version_id` and `last_seen_version_id` both set to current version, and insert the observation.
3. **Note:** Per Open Question 2 of the PRD, dedup is exact normalized-string match for v1. Fuzzy matches are deferred to operator-driven merge actions in Task 4.
4. **Note:** Observations are never moved between candidates by this script. If a candidate is `merged`, the original observations stay attached to the original (history preserved); new observations attach to the merge target.

## Requirements

1. Script lives at `peddlerr/workbench/scripts/record_candidate_terms.py`.
2. Accepts `--source-system` (or matching argument) and operates on the current `is_current=true` semantic model version for that source.
3. Idempotent on re-run: zero new rows, zero state transitions.
4. Creates new candidate rows with `status='candidate'`, both `first_seen_version_id` and `last_seen_version_id` set to current version.
5. On re-observation of a known candidate, advances `last_seen_version_id` only — does not modify status, normalized form, or notes.
6. Normalization logic is centralized in a single helper function and is documented inline.
7. Logs a structured summary on completion: `created`, `re_observed`, `skipped` counts.
