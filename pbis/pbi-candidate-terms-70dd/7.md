# 7 Purge Adena-derived canonical language for clean-slate validation

Back to task list: `tasks.md`

## Overview

Purge all Adena-derived canonical-language state so that Tasks 8 and 9 can exercise the new pipeline from a true zero state. Both Adena and System B will enter Task 9 from **identical** pre-state: their source vocabulary is preserved as a fact, but nothing in `semantic_modeling`, and only seed-root concepts in `core_semantics.canonical_concepts`. This is the only way the cross-source convergence story can be honestly demonstrated — every canonical concept that exists post-Task-9 must have arrived via the new machinery.

This task replaces the original "backfill 38 Adena deferrals" plan. The 38 deferrals were prior-process artifacts; backfilling them would smuggle pre-machinery judgments back into the new system. The cleaner test is: purge them, and let the new pipeline re-derive (or not re-derive) the same decisions through audited machinery.

### What if we didn't do this step?

**Immediate Impact:** Task 8 would run against a polluted state. Q8 would show prior Adena candidates as `candidate`/`deferred`/`promoted` even though those decisions never went through the new pipeline. The validation in Tasks 8 and 9 would be hollow — we'd be measuring metadata, not machinery.

**Downstream Consequences:** Cross-source convergence in Task 9 cannot be honestly demonstrated if half the convergent terms are already canonical concepts from a different process. The PBI's value proposition fails its own acceptance test.

**Technical Debt:** Two parallel histories (the prior ad-hoc process and the new machinery) coexist in `core_semantics.canonical_concepts`, indistinguishable from each other. Future audits cannot determine which concepts were earned and which were grandfathered.

**Alternative Approaches:** Backfill prior decisions as `status='promoted'` / `status='deferred'` candidate_terms with synthetic provenance. Rejected — it gives prior decisions the metadata of having gone through the machinery without actually exercising it.

**Risk Assessment:** Medium — destructive operation against `canonical_concepts` and `mapping_definitions`. Reversible only by re-running the prior ad-hoc process. Operator confirmation required before execution.

## Description

Author one operational script `peddlerr/workbench/scripts/reset_canonical_language.py` that:

1. **Snapshots current state** — counts and (optionally) JSON-dumps every row that will be deleted. Snapshot is written to a timestamped file under `peddlerr/workbench/scripts/data/` for audit.
2. **Asserts pre-conditions:**
   - Every `core_semantics.canonical_concepts` row with `parent_concept_id IS NULL` is in the known seed-root set: `location_type`, `org_type`, `person_role`, `healthcare_offering`, `entity_relationship`. If any other root exists, halt and report.
   - No `core_semantics.mapping_definitions` references vocabulary_terms from sources other than `adena-health-system`. If they do, halt and report (this PBI assumes Adena is the only ingested source).
3. **Prints the purge plan** — table of {table_name, count_to_delete} — and waits for `--confirm`. No `--confirm` → exit without writes.
4. **Executes the purge in one transaction:**
   - `DELETE FROM core_semantics.mapping_definitions` (all rows).
   - `DELETE FROM core_semantics.canonical_concepts WHERE parent_concept_id IS NOT NULL` (everything except seed roots).
   - `TRUNCATE semantic_modeling.candidate_terms RESTART IDENTITY CASCADE` (which also truncates `candidate_term_observations` via FK).
5. **Verifies post-state:**
   - `canonical_concepts` count = number of seed roots (should be 5 absent surprises).
   - `mapping_definitions` count = 0.
   - `candidate_terms` count = 0; `candidate_term_observations` count = 0.
   - `core_semantics.semantic_model_versions`, `source_vocabulary_models`, `vocabulary_surfaces`, `vocabulary_terms` all unchanged (Adena's published vocabulary is a fact, not derived state).
6. **Logs a final summary** with row deltas and the path to the snapshot file.

The script is **one-shot** — designed to be run once at the start of Task 7's execution and never again unless an operator explicitly chooses to reset. It is committed for auditability and reproducibility, not as runtime infrastructure.
