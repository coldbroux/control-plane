# 5 Obsolete sweep job

Back to task list: `tasks.md`

## Overview

Author a maintenance script `peddlerr/workbench/scripts/sweep_obsolete_candidates.py` that transitions deferred candidates to `status='obsolete'` when all of their observations reference superseded source vocabulary versions. Runs separately from Phase 2 — not bundled into the recording or promotion scripts.

### What if we didn't do this step?

**Immediate Impact:** Deferred candidates whose source vocabulary has been superseded persist in the active set forever, polluting Q8 review.

**Downstream Consequences:** Operators wading through the deferred list would have to mentally filter stale entries.

**Technical Debt:** The lifecycle's `obsolete` state never gets used.

**Alternative Approaches:** Trigger obsolete transitions inside `record_candidate_terms.py`. Rejected — keeps recording focused on observation writes, no cross-cutting logic.

**Risk Assessment:** Low — write-only transition with a clearly defined precondition.

## Description

The sweep walks every `candidate_terms` row with `status='deferred'`. For each, it queries the joined observations and checks whether any observation's `version_id` corresponds to a `core_semantics.semantic_model_versions.is_current=true` row. If none do, the candidate transitions:

- `status` → `'obsolete'`
- `evaluation_notes` → appended with a sweep timestamp + reason (e.g. `"obsolete sweep 2026-04-25: no observations reference is_current versions"`).
- `resolved_at` is **not** set (per CHECK constraint, `obsolete` requires resolution columns to be NULL).

The sweep does not promote or merge anything — only deferred → obsolete transitions.

The script is idempotent: re-running it produces no new transitions if no version flips occurred between runs.
